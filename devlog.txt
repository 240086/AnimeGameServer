2026-03-13

Dispatcher architecture improvement.

Problem:
MessageDispatcher was dispatching handlers through ThreadPool.
However player logic is already serialized through PlayerCommandQueue.

This resulted in double scheduling:

IO Thread
 → ThreadPool
 → PlayerCommandQueue

Fix:
Dispatcher now directly invokes handlers.

Services are responsible for pushing commands into player queues.

Benefits:
- reduced scheduling overhead
- cleaner actor-style architecture
- improved cache locality
- consistent player logic serialization

2026-03-13

Service layer refactor after Session introduction.

Problem:
Services were retrieving Player through Connection::GetPlayerId().
However Connection no longer owns Player after Session layer was introduced.

Fix:
Services now retrieve Player through Session.

Connection
   ↓
Session
   ↓
Player

Additionally removed redundant player mutex usage because
PlayerCommandQueue already guarantees serialized execution.

Architecture now follows actor-style player logic execution.
